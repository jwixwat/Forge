# v0.3 Deterministic Grading and Observation Extraction

Status: `implemented`  
Date: `2026-03-06`

## Purpose

v0.3 turns the v0.2 content contract into executable runtime behavior for deterministic formats.

This stage implements:

- deterministic grading for `resp_slots_v1` and `resp_mcq_v1`
- response canonicalization from typed response-schema steps
- raw-response shape validation separated from canonicalization and value-constraint checks
- observation extraction from rubric emission contracts
- generic runtime `obs_key` materialization with backwards-compatible `slot_pattern` support
- authoritative per-attempt `measurement_frame` binding for surface/projection/vocab semantics
- unified runtime context wiring for content registry, observation vocab, and contract validation
- rubric ambiguity tracking through `rubric_path_count`

This stage does not implement:

- LLM fallback parsing/grading
- calibration metrics or alarms
- routing or belief updates
- holdout/anchor/shadow execution policy

## Runtime Surface

Primary entry point:

- `grade_item_response(content_ir_registry, content_ir_version, item_id, raw_response, runtime_context)`
- `ForgeRuntimeContext.from_bundle(bundle)` for aligned grading + vocab + validation

It resolves the item and its declared measurement surface, then executes:

1. response canonicalization
2. deterministic rubric scoring
3. observation emission
4. grading-signal and residual-primitive materialization

Return object:

- `AttemptObservationMaterialization`
  - `measurement_subject`
  - `measurement_frame`
  - `measurement_execution`
  - `measurement_adjudication`
  - `canonical_response`
  - `grader_output`
  - `observation_result`
  - `grading_signals`
  - `residual_primitive_inputs`

`measurement_frame` now pins:

- `measurement_surface_id`
- `calibration_projection_id`
- `response_schema_id`
- `rubric_id`
- `observation_schema_id`
- `obs_encoder_version`
- `hypothesis_space_hash`
- `evidence_channel`
- `assistance_mode`
- `diagnosis_update_eligibility`
- `ineligibility_reason`

`measurement_subject` identifies what was actually graded:

- current v0.3 static path:
  - `subject_kind = "static_item"`
  - `item_id`
- current generated-instance path:
  - `subject_kind = "generated_instance"`
  - `item_instance_id`
  - `generator_id`
  - `generator_version`
  - `generator_seed`
  - `rendered_payload_hash`

`measurement_execution` records attempt-time execution facts that are not part of the
precommit-safe frame:

- `feedback_mode_applied`
- `hint_count_used`

`measurement_adjudication` is post-observation and not part of the precommit-safe frame:

- `calibration_eligible`
- `calibration_inclusion_reason`

## Deterministic Formats

### Slots

- canonicalizes slot strings using the declared response-schema steps
- grades against `items[].grading_material.slot_answer_key`
- emits:
  - `slot_pattern`
  - `slot_error_mask`
  - `latency_bucket`
  - `hint_used`

Current slot outcomes:

- `all_correct`
- `partial`
- `all_incorrect`

### MCQ

- canonicalizes `choice` using the declared response-schema steps
- grades against `items[].grading_material.correct_choice_id`
- validates the response against the authored `allowed_choice_ids` set before treating it as schema-valid
- emits:
  - `mcq_outcome`
  - `selected_choice_id`
  - `latency_bucket`
  - `hint_used`

Current MCQ outcomes:

- `correct`
- `incorrect`

## Observation Key Contract

v0.3 introduces a generic runtime `obs_key`.

- `obs_key` is derived from the bound calibration projection and emitted outcome surface inside the materialized `measurement_frame`.
- current regex starter bundle yields:
  - `slot_pattern=all_correct`
  - `slot_pattern=partial`
  - `slot_pattern=all_incorrect`
  - `mcq_outcome=correct`
  - `mcq_outcome=incorrect`

Backwards compatibility:

- slot observations still include legacy `slot_pattern`
- the v0.1 contract validator now accepts either:
  - `observation.obs_key`
  - or legacy `observation.slot_pattern`

## Ambiguity and Invalidity

v0.3 does not silently collapse malformed responses into ordinary diagnostic observations.

- `grading_signals.schema_valid` reflects canonicalization + rubric validity
- `grading_signals.rubric_path_count` is the count of accepted scoring paths
- `grading_signals.scoring_resolution_status` distinguishes:
  - `valid`
  - `invalid`
  - `unsupported`
  - reserved future states such as `ambiguous` and `abstained`
- `candidate_path_count` and `accepted_path_count` are emitted alongside `rubric_path_count`
- malformed or unsupported responses materialize:
  - `observation_status != "valid"`
  - `observation_invalid_reason`
  - no normal diagnostic `obs_key`

This gives later residual stages a mechanical signal for `spec_underdetermined` or schema-invalid cases.

`attempt.observation` should now be read as an observation envelope:

- if `observation_status == "valid"`, it is a valid instance of the bound observation schema
- otherwise, it is a non-valid envelope carrying failure/abstention status and reason

## Observation Vocabulary Bridge

`ObservationVocabularyRegistry` now supports:

- direct vocab registration by pinned binding
- bundle registration from v0.2 measurement surfaces and calibration projections

For current starter surfaces, bundle registration derives vocab from the declared calibration projection over the observation schema.

## Runtime Binding

v0.3 now carries a canonical `measurement_frame` on the attempt/precommit bridge when content-backed grading is used.

- the frame pins:
  - `measurement_surface_id`
  - `calibration_projection_id`
  - `response_schema_id`
  - `rubric_id`
  - `observation_schema_id`
  - `obs_encoder_version`
  - `hypothesis_space_hash`
- `ContractValidator` uses this frame, when present, to validate predicted observation distributions against the correct vocab binding rather than relying only on manifest-level defaults
- the validator also mirrors surface-level runtime checks:
  - emitted features must belong to the bound observation schema
  - required features must be present for valid observations
  - outcome-surface features must be present for valid observations
  - the emitted `obs_key` must match the bound calibration projection
  - emitted feature values must satisfy the bound observation schema value types and allowed-values constraints

For content-backed v0.3+ engine-generated attempts/precommits:

- `measurement_frame` is the authoritative semantic binding
- `measurement_subject` is the authoritative presented-subject binding
- `measurement_execution` is the authoritative attempt-time execution record for
  channel-constraint enforcement
- manifest-level observation binding fields remain as replay/default compatibility fields, not the source of truth

Runtime validation also enforces execution semantics against the family/channel contract:

- applied feedback mode must match the declared channel constraint
- hint usage must respect `hints_allowed` and `max_hints`
- `hint_count_used` must agree with emitted `hint_used`
- feedback traces are forbidden when the applied mode is `none`
