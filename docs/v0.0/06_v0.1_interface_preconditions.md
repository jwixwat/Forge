# v0.1 Interface Preconditions

Version: `0.0.0`  
Effective date: `2026-02-27`

## 1) Purpose

Freeze the minimum data contract that v0.1 must implement for attempt ledger and state persistence.

This prevents early schema drift and preserves compatibility with:

- evidence channel semantics
- auditable residual computation inputs
- exploration support logging
- version-linked replay
- calibration-governor diagnosis control traceability
- policy-decision operationalization support

## 2) Frozen Enums

`evidence_channel` values:

- `A_anchor`
- `B_measurement`
- `C_learning`
- `D_shadow`

`assistance_mode` values:

- `closed_book`
- `open_book`
- `tool_assisted`
- `mixed`

Identifier and timestamp integrity baseline:

- all identity fields (`run_id`, `timeline_id`, `session_id`, `attempt_id`, `precommit_event_id`, `snapshot_id`, `update_id`, `learner_id`, `item_id`, `probe_family_id`, `commitment_id`) must be non-empty strings.
- records validated against a manifest must satisfy `record.run_id == manifest.run_id`.
- all `*_ts_utc` fields must be strict RFC3339 UTC strings.

Manifest lineage baseline:

- `run_id` is a semantic epoch id.
- `timeline_id` is the longitudinal learner-continuity id.
- initial epoch (`epoch_index = 1`) must have null predecessor/bootstrap/migration fields.
- non-initial epochs must provide predecessor/bootstrap/migration linkage.

Policy-trace validity rules:

- `candidate_actions` must be non-empty.
- `candidate_actions[].action_id` must be unique within a trace.

- `chosen_action_id` must exist in `candidate_actions`.
- `chosen_action_probability` must equal the chosen candidate's logged `p`.
- `decision_traces.decision_id` must exist in `policy_decisions[].decision_id`.
- `decision_traces.trace_kind` and joined `policy_decisions.policy_domain` must be semantically compatible (for core kinds: routing/holdout/anchor/diagnosis/calibration exact-match; feedback maps to `other`; quarantine maps to `invariance|other`; `other` maps to `other`).
- `trace_kind` must be one of the frozen values above (free-form trace kinds are invalid).

- `policy_decisions` (array; may be empty)
- each element requires:
  - `decision_id` (string)

  - `rule_id` (string)
  - `scope_type` (`attempt` | `item` | `family` | `stratum` | `session` | `run` | `learner`)

  - `commit_status` (`committed` | `dropped` | `failed`)
  - `reason_code` (string)
  - `decision_ts_utc` (RFC3339 timestamp)
  - `entropy_floor_met` (bool or null)
  - `min_support_met` (bool or null)
  - `support_check_status` (`pass` | `fail` | `not_applicable`)

Policy decision derivation rule:

- `policy_applied(decision)` is defined as `decision.outcome == "applied"` and `decision.commit_status == "committed"`.

OPE support staging note:

- OPE readiness classification is run-level (`ope_support_level`) and resolved from the run manifest.
- run manifest must include `ope_claim_contract`:
  - `target_trace_kinds` (array; required, non-empty)
  - `context_axes` (array; required, non-empty; from `probe_family_id | assistance_mode_derived | evidence_channel | session_id | learner_id`)
  - `min_stochastic_fraction` (number in `[0,1]`)
  - `min_candidate_probability` (number in `(0,1]`)
  - `min_chosen_probability` (number in `(0,1]`)
  - `min_entropy_bits` (number in `[0, +inf)`)
  - `min_context_coverage_fraction` (number in `[0,1]`)
  - `min_decisions_per_context` (int `>=1`)
- `full_support` classification is target-kind and threshold based, not "any stochastic trace exists."
- `propensity_only` means: valid claim contract plus at least one valid propensity-logged decision trace for each target trace kind; OPE-readiness thresholds are not asserted.
- support-check claims are mechanical, not trusted assertions: for each targeted decision, `entropy_floor_met`, `min_support_met`, and `support_check_status` must match recomputation from `candidate_actions` against manifest thresholds.

Prequential commitment rule:

- `decision_traces`, `policy_decisions`, and `likelihood_sketch` are prequential artifacts.
- these artifacts must be committed in a prior `attempt_precommitted` record referenced by:
  - `precommit_event_id`
  - `precommit_hash`
  - `precommit_envelope_hash`
- `attempt_observed` must bind to that prior record and cannot alter these artifacts post-hoc.

Required diagnosis-governance fields:

- `diagnosis_update_eligibility` (`eligible` | `ineligible`)
- `ineligibility_reason` (`none` | `assistance_mode_not_closed_book` | `channel_not_diagnostic` | `assistance_and_channel`)

Pre-v2.0 policy default:

- if `diagnosis_update_eligibility = ineligible`, then `allowed_update_partitions` must not include `diagnosis_state`.
- diagnosis updates are valid only when `diagnosis_state` is authorized in `allowed_update_partitions` and closed-book diagnostic eligibility rules are satisfied.
- diagnosis eligibility is computed from `assistance_mode_derived` + `evidence_channel` (not from self-reported assistance mode).
- if `assistance_derivation_quality != derived_from_telemetry`, attempt must be ineligible for diagnosis updates.

### 3.1 Attempt Precommit Record Contract

Required fields for each `attempt_precommitted` record:

- `schema_version`
- `run_id`
- `timeline_id`
- `session_id`
- `precommit_event_id`
- `attempt_id`
- `learner_id`
- `presented_ts_utc`
- `item_id`
- `probe_family_id`
- `commitment_id`
- `evidence_channel_intended`
- `assistance_contract_intended`
- `diagnosis_update_eligibility_intended`
- `ineligibility_reason_intended`
- `allowed_update_partitions_intended`
- `telemetry_event_ids_intended`
- `decision_traces`
- `policy_decisions`
- `likelihood_sketch`
- `version_pointers`
- `precommit_hash`
- `precommit_envelope_hash`

Precommit/observation ordering rule:

- for each `attempt_observed`, there must exist exactly one matching `attempt_precommitted` by `(run_id, attempt_id, precommit_event_id)`.
- `attempt_precommitted.presented_ts_utc <= attempt_observed.attempt_ts_utc`.
- `attempt_observed.precommit_hash == attempt_precommitted.precommit_hash`.
- `attempt_observed.precommit_envelope_hash == attempt_precommitted.precommit_envelope_hash`.
- identity-critical bindings must match between precommit and observed attempt:
  - `run_id`, `session_id`, `attempt_id`, `learner_id`, `item_id`, `probe_family_id`, `commitment_id`.
- semantic-intent bindings:
  - `attempt_precommitted.evidence_channel_intended == attempt_observed.evidence_channel`.
  - `attempt_observed.assistance_mode` may be equal-or-more-assisted than `assistance_contract_intended`, never less-assisted.
  - intended diagnosis fields must be internally coherent with `(assistance_contract_intended, evidence_channel_intended)`.
  - observed diagnosis fields must be internally coherent with `(assistance_mode, evidence_channel)`.
- `attempt_precommitted.telemetry_event_ids_intended` must match `attempt_observed.telemetry_event_ids`.
- telemetry window completeness is required: the declared `attempt_observed.telemetry_event_ids` must equal the full committed telemetry set for `(run_id, timeline_id, attempt_id, learner_id)` in `[presented_ts_utc, attempt_ts_utc]`.
- every referenced telemetry event must belong to the same `(run_id, timeline_id, attempt_id, learner_id)` and must be within `[presented_ts_utc, attempt_ts_utc]`.
- telemetry window must include at least one `response_submitted` event and event-ledger ordering must satisfy: `attempt_precommitted` precedes `response_submitted`, which precedes/equal `attempt_observed`.

Event-ledger append-time authority:

- each committed event carries immutable `event_written_ts_utc` assigned at append time.
- sequence coherence and precommit authenticity are evaluated using `ledger_sequence_no` + `event_written_ts_utc`.
- payload timestamps remain diagnostic metadata and anti-time-travel checks; they are not authoritative for causal ordering.

### 4.1 State Update Event Contract (Frozen for v1.1 Enforcement)

Required fields for each `state_update_event`:

- `schema_version`
- `run_id`
- `timeline_id`
- `session_id`
- `snapshot_id`
- `update_id`
- `learner_id`
- `update_ts_utc`
- `source_attempt_id`
- `stratum_id`
- `calibration_status_at_update` (`healthy` | `warning` | `miscalibrated_persistent`)
- `governor_decision` (`normal` | `throttle` | `strong_throttle` | `freeze`)
- `governor_reason`
- `applied_update_multiplier` (number)
- `safe_mode_profile_id`
- `governor_transform_version` (string)
- `state_patch` (object with authoritative mutation target in `state_patch.partition`)
- `proposed_state_patch` (object; deterministic pre-governor proposal)
- `base_value_at_proposal` (number)
- `pre_state_hash` (string)
- `post_state_hash` (string)
- `diagnosis_log_write_status` (`committed` | `failed` | `missing`)
- `mutation_attempted` (bool)
- `mutation_outcome` (`applied` | `blocked_by_governor` | `skipped_by_policy` | `failed_due_to_integrity`)
- `suppression_reason` (`none` | `calibration_freeze` | `policy_skip` | `quarantine` | `other`)
- `log_commit_id` (string or null)
- `mutation_applied` (bool)
- `integrity_event_id` (string or null)

Status semantics note:

- `diagnosis_log_write_status` refers to durability status of the diagnosis-mutation log channel, not the append status of the event ledger itself.
- state update events are always event-ledger records; this field governs whether diagnosis-state mutation authority is granted.

Governor multiplier ladder for diagnosis-state updates:

- `normal -> 1.00`
- `throttle -> 0.50`
- `strong_throttle -> 0.20`
- `freeze -> 0.00`

State-update governor rules:

- If `state_patch.partition = diagnosis_state`, all governor fields above are required and non-null.
- If `safe_mode_profile_id = SG_CALIBRATION_GUARD`, `applied_update_multiplier` must equal the multiplier mapped from `governor_decision`.
- If `safe_mode_profile_id = SG_SENSOR_UNRELIABLE_GUARD`, `applied_update_multiplier` must be `0.10`.
- If `safe_mode_profile_id` is any `SP_*` panic profile, `applied_update_multiplier` must be `0.00`.
- If `calibration_status_at_update = miscalibrated_persistent`, `governor_decision` must be `throttle`, `strong_throttle`, or `freeze`.
- For diagnosis-state `set` patches, bounded movement is mechanically defined by:
  - `applied_value = base_value_at_proposal + m * (proposed_value - base_value_at_proposal)` where `m = applied_update_multiplier` (with profile/governor constraints above).
- `proposed_state_patch` and `state_patch` must share `op`, `partition`, and `path`.
- `freeze` must not apply diagnosis mutation (`mutation_outcome` must not be `applied`).
- Any `SP_*` panic profile must not apply diagnosis mutation (`mutation_outcome` must not be `applied`).
- If `state_patch.partition = diagnosis_state` and `diagnosis_log_write_status = committed`, mutation authority is governed by `mutation_outcome` (`applied` vs policy/governor suppression outcomes).
- If `state_patch.partition = diagnosis_state` and `diagnosis_log_write_status in {failed, missing}`, then `mutation_applied` must be `false`, `mutation_outcome` must be `failed_due_to_integrity`, and `integrity_event_id` must be non-null.
- No diagnosis-state mutation is valid if required log-write fields are missing.
- `safe_mode_profile_id` must be a known profile id from the safe-mode profile ontology.
- when a safe-mode transition profile is active in event order, each `state_update.safe_mode_profile_id` must match the active transition profile id.
- `snapshot_id` is a lineage reference id, not presentation metadata:
  - must reference a known lineage node (`__genesis__` allowed),
  - unknown lineage refs are invalid.

## 5) Idempotency and Replay Preconditions

v0.1 implementation must satisfy:

1. `(run_id, attempt_id)` uniqueness.
2. Replaying same attempt does not create duplicate state updates.
3. State update lineage is traceable via `source_attempt_ids`.
4. Replay uses manifest-linked versions from `version_pointers`.
5. `(run_id, update_id)` uniqueness for `state_update_event` records.
6. Replaying failed/missing diagnosis-state log-write events must not produce diagnosis-state mutation.
7. Event ledger assigns authoritative monotonic `ledger_sequence_no` per `run_id`.
8. Replay ordering is by `ledger_sequence_no` only; `event_ts_utc` is diagnostic metadata and must not define replay order.
9. Canonical event ledger input is required for v0.1 gate evaluation; typed record lists are auxiliary and may be used only for event-vs-typed equivalence checks.

Epoch lineage and migration preconditions:

10. Non-initial epochs (`epoch_index > 1`) require exactly one `state_migration` event bound to manifest linkage fields.
11. Required migration event must occur before first `attempt_precommitted` event in the run.
12. Event ids are type-namespaced by construction (`evt_<event_type>_...`) and must use the declared event-type prefix.
13. If both event ledger and typed record lists are supplied, they must be equivalent views: same ids and same payload hashes for each typed record class.
14. Cross-epoch state continuity is valid only via explicit `state_migration` event, never via silent snapshot carryover.

Timestamp threat-model note:

- anti-time-travel checks (precommit/telemetry/snapshot timestamp constraints) assume honest logger clocks for v0.1.
- stronger authority (trusted append-time or monotonic attestation) is a vNext hardening path and not yet enforced.

## 6) Schema Evolution Rule

Until v0.2:

- field names above are immutable
- only additive optional fields are allowed
- enum value additions require charter update and minor version bump
- validator dispatch is keyed by `schema_version` per record type (`manifest`, `attempt`, `attempt_precommit`, `attempt_telemetry`, `state_snapshot`, `state_update`, `state_migration`, governance events).
- unsupported `schema_version` values are hard-invalid at ingest/replay.
- replay normalization/upconversion targets `0.1.0`; currently this includes compatibility uplift for `manifest.schema_version=0.0.0`.
